Falsifying Reachset Conformance in Cyber-Physical Systems using Bayesian Optimization

Zeqi Li · March 7, 2024 · 12 min read

#BayesianOptimization #ReachabilityAnalysis #CyberPhysicalSystems #SafetyVerification

The safety of Cyber-Physical Systems (CPS)—such as autonomous vehicles, smart factories, and robotics—has always been a critical issue of great concern. Because these systems involve tight coupling between physical dynamics and computational algorithms, a breach in safety could lead to personal injury, financial loss, or worse.

Traditionally, we rely on verification-based methodologies like model checking, non-standard analysis, or deductive verification. However, as system architectures become increasingly complex and complete mathematical models become inaccessible, these traditional methods struggle to scale.

Enter Search-Based Testing. By treating falsification (the act of finding a counterexample that breaks system safety) as an optimization problem, we can smartly explore the state space. In this post, we’ll explore a Robustness-Guided Search utilizing Bayesian Optimization (BO). We will use this method to falsify Reachset Conformance between an abstract kinematic model and a highly complex dynamic vehicle model.

1. Reachability Analysis & Conformance Testing

What is a Reachable Set?

In a dynamical system, a reachable set contains all the possible states a system can reach at a certain moment $t$, given a set of initial states and inputs. Because natural systems (like autonomous vehicles) are non-deterministic—plagued by control uncertainty and environmental disturbances—we must account for non-deterministic terms $\omega \in \mathcal{W}$.

The reachable set $\mathcal{R}^{non}(u)$ extended by non-determinism is formulated using the Minkowski sum ($\oplus$):

$$\mathcal{R}^{non}(u)=\left\{x(t) \mid \dot{x}(t)\in f(\mathcal{X}(k),\mathcal{U}(k))\oplus\mathcal{W}, x(0)\in\mathcal{X}^0, u(t)\in\mathcal{U}, \omega\in\mathcal{W}\right\}$$

(Note: The Minkowski sum adds every point vector in the uncertainty set $\mathcal{W}$ to every point in the system's state derivatives, creating a bounding envelope of all possible physical realities).

To compute and represent these sets efficiently, we use Zonotopes.

Definition: Zonotope

Given a center vector $c\in\mathbb{R}^n$ and a generator matrix $G\in\mathbb{R}^{n\times q}$, a zonotope is defined as:

$$\mathcal{Z} = \left\{ c + \sum_{j=1}^{q} \beta_j G_{(\cdot, j)}\ \middle|\ \beta_j \in \left[-1,1\right] \right\}$$

Zonotopes are brilliant for this because they are closed under linear transformations and naturally represent bounds (measurement noise, external disturbances) modeled as intervals.

Reachset Conformance

To plan motion efficiently, autonomous agents use a simplified Abstract Model (a white-box system). But we must guarantee that this simplified model is a safe representation of the Complex Model (the real, highly complex, black-box vehicle).

Reachset Conformance dictates that if the true trajectory of the complex model is always contained within the reachable set of the abstract model, the complex model is conformant to the abstract model:

$$\forall t: \forall u\in\mathcal{U}: Trace_t(S_c,u)\subseteq Reach_t(S_a,u)\Leftrightarrow S_c\mathrm{~conf}_{\mathcal{R~}}S_a$$

Our goal is Falsification: finding a specific test case (input sequence) where the complex trajectory breaches the boundaries of the abstract reachable set.

2. Guiding the Search: Bayesian Optimization

Because simulating complex dynamic models is computationally expensive, we cannot use brute-force grid search. Gradient descent is also out of the picture because our system is a black-box and gradients aren't available (and it's prone to local minima).

We use Bayesian Optimization (BO), an algorithm explicitly designed for black-box, derivative-free global optimization.

Algorithm 1: Bayesian Optimization
---------------------------------------------------------
Require: Input domain X; Initial dataset D_0
1: GP_MODEL <- GP_REGRESSION(D_0)
2: for t = 1, ..., N do
3:     x_t <- argmax_{x \in X} AC(GP_MODEL(x))
4:     y_t <- OB(x_t)
5:     D_t <- D_{t-1} U {(x_t, y_t)}
6:     GP_MODEL <- GP_REGRESSION(D_t)
7: end for
8: return D_N

BO consists of two critical components:

A. Gaussian Process Regression (GPR)

GPR acts as the surrogate model fitting our objective function. Instead of outputting a single value, GPR models the entire distribution of possible outputs, uniquely determined by a mean function $\mu(x)$ and a covariance (kernel) function $K(x, x')$. This provides not just a prediction, but a measure of uncertainty ($\sigma(x)$).

B. Acquisition Function (AC)

The Acquisition Function uses the GP's posterior distribution to score which test case $x$ we should sample next, balancing exploration (high variance/uncertainty) and exploitation (low mean/known good results). We evaluate three distinct functions:

  1. Upper Confidence Bound (UCB): $UCB(x) = \mu(x) + \kappa \sigma(x)$
  2. Probability of Improvement (PI): Focuses on the probability that a new sample will improve over the best current observation.
  3. Expected Improvement (EI): Quantifies the expected magnitude of improvement.

3. Modeling the Vehicles & Defining the Objective

To perform our tests, we define two mathematical models.

The Abstract Model (Kinematic Single-Track)

This is the "Bicycle Model". It simplifies four wheels into two and focuses on steering dynamics, ignoring complex slip angles. It is governed by:

$$\begin{aligned} \dot{x}_1 &= x_{4}\cos(\beta+x_{5}) + \omega_1 \\ \dot{x}_2 &= x_{4}\sin(\beta+x_{5}) + \omega_2 \\ \dot{x}_3 &= 0.98 * u_{1} + \omega_3 \\ \dot{x}_4 &= 0.98 * u_{2} + \omega_4 \\ \dot{x}_5 &= x_{4}\cos(\beta)\frac{\tan\left(x_{3}\right)}{l} + \omega_5 \end{aligned}$$

Where the state vector $\textbf{x}=[p_x,p_y,\delta,v,\psi]^\top$ tracks position, steering angle, velocity, and yaw angle. To compute its reachable sets, we discretize the system using the Taylor series inside MATLAB's Continuous Reachability Analyzer (CORA) toolbox.

Parameterizing the Test Cases

To optimize effectively, we parameterize the steering angle input $u(t)$ into geometric curves. We explored three types:

  1. Sinusoid Curve: Governed by Amplitude $A$, Period $T$, and Shift $T_s$.
  2. Bézier Curve: Uses polynomial control points ($p_1, p_2, p_3$).
  3. Sigmoid Curve: Creates S-shaped steering transitions.

The Objective Function: Robustness

The "score" BO is trying to minimize is Robustness. We define robustness as the minimum Euclidean distance between the complex model's trajectory points $\mathcal{T}_C^{t_k}(u)$ and the abstract model's reachable set boundary $\mathcal{R}_A^{t_k}(u)$ across all time steps.

We also account for measurement error $\varepsilon$ using an infimum distance formula:

$$dist(x,\mathcal{R}) = \begin{cases} \inf\{d(x,y)\mid y\in\mathcal{X}\backslash\mathcal{R}\}-\varepsilon & \mathrm{~if~}x\in\mathcal{R}\\ -\inf\{d(x,y)\mid y\in\mathcal{R}\}-\varepsilon & \mathrm{~if~}x\notin\mathcal{R} \end{cases}$$
  • ► Positive Robustness: The trajectory is safely inside the reachable set.
  • ► Negative Robustness: The trajectory breached the reachable set (Falsification achieved!).

4. Simulation Results & Discussion

Unit Testing Bayesian Optimization

Because BO relies heavily on prior knowledge and complex updates, we first validated our architecture using a Unit Test. We ran the system on a simplified linear model:

  • Complex Linear: 3 states ($\dot{x}(t) = A_{3\times3}x + B u$)
  • Abstract Linear: 2 states ($\dot{x}(t) = A_{2\times2}x + B u$)

The code successfully verified modifications to the CORA backend, bounding robustness values tightly within $[-0.25, 0.25]$ and confirming BO’s ability to find negative robustness.

Efficacy of the Search

Transitioning to the non-linear vehicle models, we initialized BO with a 5-sample dataset using the Sinusoid curve:

Test case A T $T_s$ v Robustness
1 6 7 2 3 $2.6099 \times 10^{-5}$

Using the UCB acquisition function with a 50-iteration limit, BO efficiently navigated the search space. By the 25th iteration, BO discovered a set of parameters [7.99, 4.00, 3.99, 5.99] that drove the robustness value to $-0.2587$. We successfully falsified the model!

Comparing Acquisition Functions

The choice of acquisition function drastically impacts performance:

  • UCB (Upper Confidence Bound): Successfully found the global minimum by cleanly balancing mean and variance.
  • EI (Expected Improvement): Exhibited high fluctuation. It required bumping iterations up to 100 to finally identify the global minimum.
  • PI (Probability of Improvement): Rapidly stabilized early in the run but got trapped in a local minimum, failing to find the true falsification case regardless of extra iterations.

Furthermore, swapping the Sinusoid curve for Bézier and Sigmoid curves yielded similarly successful falsifications within 50 iterations, proving the flexibility of the parameterization.

Generalizability: Semi-Trailer Truck Model

To ensure this methodology scales to highly complex industry use-cases, we replaced the simple car with a Semi-trailer truck model (Tractor, Hitch, and Trailer).

The abstract representation tracks planar motion and hitch angle $\alpha$. The complex counterpart is a Simulink 3DOF Vehicle Dynamics Blockset model that accounts for aerodynamic drag $F_{\mathrm{aero}}$, rolling resistance $F_{\mathrm{roll}}$, and requires a closed-loop forward feedback controller to align torque $T$ inputs with the abstract model's acceleration.

After 50 iterations, BO evaluated the individual conformance of the Tractor, Hitch, and Trailer. The algorithm successfully identified a test case where the trajectory of the Hitch narrowly exceeded its reachable set, proving the method's robust generalizability to multi-body dynamical systems.

5. Deep Dive: Why This Architecture, and Is It Reasonable?

If you look closely at the methodology, you might wonder: Why go through the trouble of combining abstract models, complex models, continuous robustness metrics, and Bayesian Optimization?

Let's break down the engineering pain points and the logical chain that makes this architecture not just reasonable, but a highly elegant solution for the industry.

The Logical Chain: Why do it this way?

1. The Abstract vs. Complex Dilemma

  • ► The Pain Point: High-fidelity vehicle models (like a 7-DOF Simulink model) are computationally heavy and essentially "black boxes." If an autonomous driving system uses this for real-time trajectory planning, the compute lag would be fatal.
  • ► The Solution: The vehicle runs a highly simplified "Bicycle Kinematic Model" (the Abstract Model) which calculates quickly and can generate mathematical "safety boundaries" (Reachable Sets).
  • ► The Catch: How do you prove your simplified model isn't too simple? What if the real vehicle's physical inertia pushes it outside those safety boundaries? Hence, we need Reachset Conformance Testing to pit the real physics against the simplified math.

2. Escaping the "Boolean Trap" with a Robustness Metric

  • ► The Pain Point: If we just check if a trajectory breaches the reachable set, the output is binary: True (safe) or False (unsafe). For a search algorithm, this is like trying to find a hole in a massive flat field while blindfolded—there is no sense of direction.
  • ► The Solution: The authors transformed a Boolean problem into a continuous distance problem. By calculating the distance from the trajectory to the safety boundary, the algorithm gets a "Robustness Score."
    • Distance > 0: Safe (but tells us how far we are from danger).
    • Distance < 0: Breach! (Falsification successful).
  • ► The Result: The search algorithm now has a gradient. It simply follows the path where the robustness number keeps getting smaller.

3. Why Bayesian Optimization?

  • ► The Pain Point: We need to find the exact input that causes a negative robustness score.
    • - Random Search is too inefficient.
    • - Grid Search causes a combinatorial explosion.
    • - Gradient Descent is impossible because the complex vehicle simulator is a derivative-free black box.
  • ► The Solution: Bayesian Optimization (BO) is custom-built for expensive, black-box functions. By building a Gaussian Process surrogate model in its "mind," BO guesses where the minimum might be while mathematically balancing the exploration of unknown areas.

4. Beating the Curse of Dimensionality via Parameterization

  • ► The Pain Point: A steering sequence over 5 seconds might contain hundreds of data points (time steps). If BO tries to optimize a 500-dimensional array, the math breaks down.
  • ► The Solution: The authors forced the steering inputs to follow geometric curves (like Sinusoid or Bézier). Instead of optimizing 500 individual points, BO only needs to tweak 3 or 4 parameters (e.g., Amplitude, Period, Shift).

The Verdict: Is this approach reasonable?

Yes, it is a highly reasonable and cutting-edge engineering trade-off.

Think of Bayesian Optimization in this context as a brilliant detective: Instead of interrogating every single suspect in a city (Grid Search), the detective interrogates a few key people, models the probability of where the culprit is hiding (Gaussian Process), and catches the suspect in just 25 to 50 attempts.

The Strengths:

  • ► Extreme Sample Efficiency: It saves massive amounts of compute time. Running a heavy Simulink trailer simulation thousands of times is unfeasible; doing it 50 times is trivial.
  • ► Black-Box Friendly: This methodology requires zero knowledge of the complex model's internal differential equations. This perfectly matches the automotive industry, where OEMs often use compiled, black-box models provided by Tier-1 suppliers.

The Inherent Risks (The "Gotchas"):

However, the architecture does have bounds:

  1. The "Flat Landscape" Trap: If the abstract model is given too much uncertainty ($\omega$), the reachable set becomes a massive envelope. The trajectory might run parallel to the boundary, resulting in a constant robustness score. If the score doesn't change, BO loses its "scent" and searches blindly.
  2. Dimensionality Limits: While parameterizing inputs into 4 dimensions works beautifully, if we were to test a highly complex urban scenario (requiring 30+ parameters for pedestrians, traffic lights, and multiple vehicles), the $O(N^3)$ matrix inversion complexity of BO's Gaussian Process would cause the algorithm itself to bottleneck.

Ultimately, by bridging formal reachability analysis with probabilistic machine learning, this framework offers a highly scalable, practical path forward for testing the un-testable in Cyber-Physical Systems.

6. Conclusion & Future Outlook

While Bayesian Optimization is sample-efficient and mathematically elegant, the research highlighted specific challenges:

  1. Flat Robustness Landscapes: If the abstract model’s non-determinism bounds ($\mathcal{W}$) are overly conservative (too large), the trajectory maintains a constant distance to the nearest set edge. This creates a flat optimization landscape (constant robustness), starving the Gaussian Process of gradients and preventing it from escaping local minima.
  2. Poor Abstract Model Design: If the abstract model fundamentally fails to capture the complex dynamics, trajectories will wildly deviate. Falsification becomes trivial, but the resulting test cases are useless for edge-case debugging.
  3. Hardware & Time Constraints: BO's GP updates scale poorly ($O(N^3)$) with dataset size. Limiting iterations prevents finding the global optimum if the space is highly non-linear.

Future Directions:
To escape deep local minima, future iterations could integrate Monte Carlo Tree Search (MCTS) as an upper-layer guide for BO, effectively combining global tree exploration with BO's local exploitation. Furthermore, integrating Bayesian Optimization natively into reachability toolboxes (like CORA) as an alternative to classical Rapidly-exploring Random Trees (RRT) presents an exciting frontier for conformance synthesis.

Conclusion

This project demonstrates that coupling Reachset Conformance checking with Robustness-Guided Bayesian Optimization is a highly effective, generalized approach for test case generation. By parameterizing inputs and treating safety verification as an optimization problem, we can systematically hunt down system vulnerabilities—ensuring that the cyber-physical systems of tomorrow are fundamentally safer.
